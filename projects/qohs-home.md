# QOHS Home — 予約・顧客管理システム + スマホアプリ

<Badge type="info" text="2020-2021年" /> <Badge type="tip" text="1人開発" /> <Badge type="warning" text="企画〜リリース" />

## 背景

QOHS（キューオーエイチエス）はリラクゼーションサロンのブランドです。QOHS事業部への配属に伴い、顧客向けのスマホアプリと、それを支えるバックエンドシステムを新規開発することになりました。

このプロジェクトが特別なのは、**Webサーバー構築からスマホアプリのUIUXデザイン、API開発、管理画面構築まで、全工程を1人で企画からリリースまで完遂した**ことです。

## 課題

- 顧客が予約や自分の情報を確認する手段がなかった
- 店舗スタッフの顧客管理が属人的で、引き継ぎが困難
- 顧客とのコミュニケーションチャネルがなかった

## 設計判断

### アーキテクチャ

```
[スマホアプリ (Cordova + Vue.js)]
        ↓ API通信
[APIサーバー (Laravel)]
        ↓
[データベース (MySQL on AWS)]
        ↑
[管理画面 (Laravel)]
```

### なぜ Cordova か

2020年当時、クロスプラットフォームの選択肢として React Native、Flutter、Cordova がありました。

- **React Native**: 学習コストが高く、1人開発では立ち上がりに時間がかかる
- **Flutter**: 当時はまだエコシステムが成熟しておらず、日本語の情報が少なかった
- **Cordova + Vue.js**: 既にVue.jsの経験が豊富で、Webの知識をそのまま活かせる

「最高の技術」ではなく「1人で最速でリリースできる技術」を選びました。ネイティブに近いパフォーマンスよりも、**リリースして顧客に価値を届けること**を優先した判断です。

### なぜ Laravel か

APIサーバーと管理画面を同じフレームワークで構築することで、開発・保守の効率を最大化しました。Laravel Passport によるOAuth2認証で、アプリとAPIの認証基盤を構築。

| 領域 | 技術 | 選定理由 |
|------|------|----------|
| インフラ | AWS + Linux | スケーラビリティとコスト管理のバランス |
| アプリ | Cordova + Vue.js | 既存のWeb知識を最大活用 |
| API/管理画面 | Laravel | 1つのフレームワークで2つの役割をカバー |
| 認証 | Laravel Passport | OAuth2ベースのセキュアな認証 |
| デザイン | Adobe XD | プロトタイプ→ユーザーテスト→修正のサイクル |

## 実装の全工程

### 1. 企画・要件定義
サロンの現場スタッフと顧客の両方にヒアリングを実施。「何が欲しいか」ではなく「普段どう過ごしているか」を聞くことで、本当のニーズを掘り出しました。

### 2. UIUXデザイン（Adobe XD）
スマホアプリと管理画面の両方をデザイン。特にスマホアプリは、サロンの顧客層（30-50代女性が中心）を意識し、**シンプルで直感的なUI**を設計しました。

### 3. インフラ構築（AWS + Linux）
Webサーバーの構築から始め、SSL証明書の設定、セキュリティグループの設計、バックアップ体制まで1人で構築。

### 4. API開発（Laravel）
RESTful APIの設計・実装。Laravel Passportによる認証基盤の構築。

### 5. 管理画面構築（Laravel）
店舗スタッフが使う管理画面を Laravel の Blade テンプレートで構築。

### 6. スマホアプリ開発（Cordova + Vue.js）
顧客向けスマホアプリの開発。プッシュ通知、予約機能、顧客情報の閲覧機能を実装。

### 7. テスト・リリース
現場でのベータテストを経てリリース。リリース後も現場の声を聞きながら継続的に改善。

## ユーザーインタビューからの改善

リリース後、最も価値があったのは**エンドユーザーへのインタビュー**でした。

開発者自身がユーザーの声を直接聞くことで、仕様書やチケットでは伝わらない「温度感」を感じ取れます。例えば：

- 「予約確認画面で次回予約の日付がわかりにくい」→ フォントサイズと色のコントラストを調整
- 「お気に入りのスタッフを指名したいけどやり方がわからない」→ 指名予約のUIフローを再設計

こうした改善は、チケットの文面からは生まれない。**現場に行って、使っている人の横に立つ**ことでしか得られない気づきです。

## 学び

**「全部やる」ことの本当の価値は、全体最適ができること。**

フロントエンドだけ、バックエンドだけ、デザインだけを担当していたら、各レイヤーの最適化しかできない。全工程を1人で経験したからこそ、「このUIを実現するためにAPIをこう設計すれば、フロントのコードがシンプルになる」「このインフラ構成なら、将来のスケールにも対応できる」といった**レイヤーを跨いだ判断**ができるようになりました。

2021年6月からは新しくジョインしたメンバーと一緒に運用・保守・機能追加を担当。1人で作ったシステムをチームで運用するフェーズも経験しました。

今振り返ると、Cordova ではなく React Native や Flutter を選んでいれば、よりリッチなUXを提供できたかもしれません。でも「1人でリリースまで持っていく」という制約の中では、正しい判断だったと確信しています。
