# TenStar — 社内業務システム

<Badge type="info" text="2020年" /> <Badge type="tip" text="1人開発" /> <Badge type="warning" text="企画〜開発" />

## 背景

株式会社SMILE CREATE GROUPは複数のリラクゼーションサロンを運営しており、各店舗の業務管理が課題でした。既存の業務フローは紙やExcelベースで、情報の一元管理ができていない状態。

「社内業務を効率化するシステムが必要だ」という要件が上がったとき、社内にエンジニアは私1人。外注するか、自分で作るか。**自分で作る**道を選びました。

## 課題

- 複数店舗の業務データが分散しており、横断的な管理ができない
- 紙・Excelベースの運用で入力ミスや情報の遅延が発生
- 既製のSaaSでは業務フローに合わない部分が多い

## 設計判断

### なぜフルスクラッチか

既製のSaaS（kintone等）も検討しましたが、リラクゼーションサロン特有の業務フロー（予約管理、施術記録、スタッフシフト等）に合わせるカスタマイズコストを考えると、フルスクラッチの方が長期的にメンテナブルだと判断しました。

### 技術選定

| 領域 | 技術 | 選定理由 |
|------|------|----------|
| バックエンド | PHP | 社内に知見があり、ホスティングコストが低い |
| サーバー | Linux | AWS上で自前構築。コスト管理が容易 |
| デザイン | 自作 | Adobe XDでプロトタイプを作成し、現場スタッフにヒアリング |

Vue.jsやLaravelも選択肢にありましたが、システムの規模感と保守性を考え、シンプルなPHPで構築する判断をしました。「使う技術を最小限にすることで、自分1人でも保守できる」という考え方です。

## 実装

1人で以下の全工程を担当しました：

1. **要件定義**: 現場スタッフへのヒアリングから業務フローを整理
2. **設計**: データベース設計、画面設計
3. **デザイン**: 管理画面のUIデザイン（Adobe XD）
4. **サーバー構築**: Linux環境のセットアップ
5. **開発**: PHP によるバックエンド実装
6. **テスト・リリース**: 現場での受け入れテストを経てリリース

## 学び

**「作れる」と「使われる」は違う。**

技術的に動くものを作ることは難しくなかった。難しかったのは、ITリテラシーが高くない現場スタッフが「迷わず使える」UIを設計すること。プロトタイプを現場に持っていき、実際に操作してもらいながら何度もUIを修正しました。

この経験から、**開発者がユーザーの近くにいることの価値**を強く実感しました。ユーザーインタビューを開発者自身がやることで、仕様書では伝わらないニュアンスを直接設計に反映できる。これは今でも大切にしている考え方です。

もし今作り直すなら、TypeScript + Next.js でフロントエンドを構築し、APIはREST or GraphQLで分離するでしょう。でも当時の判断として「1人で保守できるシンプルさ」を優先したことは正しかったと思っています。
